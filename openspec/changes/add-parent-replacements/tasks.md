## 1. Discovery & Design
- [ ] 1.1 明确自动确认规则（容量、同级别/场馆等）和人工审核条件。
- [ ] 1.2 定义补课申请数据结构与状态机，确定与现有 sessions/attendance 的关系。
- [ ] 1.3 确认通知渠道与文案（确认/拒绝/撤销）、触发点和受众。

## 2. Implementation
- [ ] 2.1 新增数据/仓库方法：查询可补课班次、创建/更新补课申请、容量与时间冲突校验。
- [ ] 2.2 家长端补课列表与筛选 UI；申请对话框（选择孩子、目标班次、备注）。
- [ ] 2.3 申请状态视图（待处理/已确认/已拒绝），支持在开课前撤销。
- [ ] 2.4 提交时的自动确认与待审核分支；确认后写入目标班次报名/出勤关联。
- [ ] 2.5 更新日程/出勤展示：显示补课场次、来源课次以及缺勤标记。
- [ ] 2.6 通知下发：补课申请的确认/拒绝/撤销通知家长（及可选管理员/教练）。

## 3. Validation
- [ ] 3.1 测试容量/冲突校验、自动确认与待审核分支。
- [ ] 3.2 端到端验证：请假 -> 搜索班次 -> 申请 -> 确认/拒绝/撤销 的用户流程。
